1. ARCHITECTURE TECHNIQUE
================================================================================

1.1 Stack Frontend
- PHP 8.2+ / Symfony 6.4+
- Templates Twig avec héritage de layout
- Composants Réutilisables (partials Twig + Stimulus.js)
- Hotwired Turbo pour navigation rapide
- Design System implémenté en SCSS avec variables CSS

1.2 Layouts
- base.php Layout racine: <html>, <head> avec assets, <body>, block content
- auth.php Layout authentification: centré, fond #F0F4F5, max-width 480px
- backoffice.php Layout principal: sidebar (256px collapsed: 72px), topbar (64px), content area, footer

1.3 Partials Réutilisables
- sidebar.php: Navigation principale, logo MicroFin, menu items avec icônes, badge notifications, user info en bas
- topbar.php: Breadcrumb, search global (Ctrl+K), notifications dropdown, user avatar/dropdown, help icon
- breadcrumb.php: Accueil > Module > Sous-module > Écran courant
- flash.php: Messages de feedback success/error/warning/info
- table-toolbar.php: Search input, filters dropdown, date range picker, export button, action buttons
- pagination.php: Prev/Next, page numbers, items per page selector (10/25/50/100)
- export-actions.php: CSV, Excel, PDF, Print buttons
- confirm-modal.php: Modal de confirmation générique avec message personnalisable
- empty-state.php: Icône + message + action button pour listes vides
- form-errors.php: Error summary en haut du formulaire
- approval-banner.php: Bandeau orange pour actions en attente d'approbation

1.4 Composants Spécifiques
- status-badge.php: Badge coloré par statut (actif=vert, inactif=gris, bloqué=rouge, en_attente=orange)
- data-table.php: Table responsive avec tri, recherche, pagination, zebra #F8FAFA
- modal.php: Modal overlay avec backdrop, taille sm/md/lg/xl
- card.php: Card avec padding 24px, border 1px #E0EAEB, radius 8px
- form-group.php: Label + input + error + help text
- autocomplete-input.php: Input avec dropdown de suggestions AJAX
- date-range-picker.php: Deux inputs date avec sélection rapide (aujourd'hui, 7j, 30j, ce mois)
- file-upload.php: Drag & drop zone + preview + progress bar
- signature-pad.php: Zone de signature canvas
- document-viewer.php: Visualisation PDF/image inline
- activity-timeline.php: Timeline verticale d'activités
- audit-trail.php: Tableau d'audit avec détails avant/après
- confirmation-flow.php: Étape de confirmation avant action critique
- loading-spinner.php: Spinner overlay pour chargements AJAX
- progress-steps.php: Indicateur d'étape pour workflows multi-step

1.5 Architecture des Données
- Toutes les données via API REST Java (api.microfin.ci)
- Cache local Symfony pour les référentiels (produits, agences, utilisateurs)
- Realtime notifications via Server-Sent Events ou polling 30s
- Export généré côté serveur, téléchargement asynchrone

1.6 Responsive Breakpoints
- Desktop: 1280px+ (sidebar expanded)
- Tablet: 768-1279px (sidebar collapsed, icônes seulement)
- Mobile: <768px (sidebar off-canvas, stacked layouts)

5. WORKFLOWS COMPLETS
================================================================================

5.1 WORKFLOW CONNEXION
----------------------------------------------------------------
1. Utilisateur accède à login.php
2. Saisit identifiant + mot de passe
3. Système vérifie: existence compte, statut (actif/bloqué/expiré), MDP valide
4. Si compte bloqué: "Votre compte est verrouillé. Réessayez dans X minutes ou contactez le support"
5. Si MDP expiré (>90 jours): redirige → expired-password.php (changement obligatoire)
6. Si MDP valide mais 2FA activée: redirige → otp-verify.php
7. Code OTP envoyé par email et/ou SMS
8. Utilisateur saisit code OTP (6 chiffres, valide 5 min)
9. Si OTP valide: session créée, redirect dashboard selon rôle
10. Si OTP invalide: message erreur, 3 tentatives → nouveau code requis
11. Si multi-agence: écran selection agence (agency-select.php)
12. Session HTTP, timeout 30min inactivité, JWT pour les appels API

5.2 WORKFLOW TRANSACTION (DÉPÔT/RETRAIT/VIREMENT)
----------------------------------------------------------------
1. GUICHETIER sélectionne type opération
2. Recherche client (autocomplete)
3. Sélectionne le compte (si plusieurs)
4. Saisit montant (XOF, >0, <= plafond agence/guichetier)
5. Sélectionne mode de paiement (espèces/chèque/virement)
6. Saisit motif (text, required)
7. Vérification: solde suffisant (retrait), plafond non dépassé, compte actif
8. Confirmation: récapitulatif (client, compte, montant, frais, net)
9. Si montant > seuil libre (500000 XOF): workflow Maker-Checker déclenché
10. Si montant > seuil supérieur (5000000 XOF): double validation requise
11. Exécution: écriture compte, journal opérations, mise à jour solde, caisse
12. Impression reçu (operation-receipt.php)
13. Notification client SMS (si option activée)
14. État: COMPLETED (ou PENDING_VALIDATION si Maker-Checker)

5.3 WORKFLOW CRÉDIT COMPLET (DEMANDE → DÉCAISSEMENT → REMBOURSEMENT)
----------------------------------------------------------------
PHASE 1: DEMANDE
1. Client soumet demande (via guichetier: credit-request-create.php)
2. System enregistre demande (statut: PENDING_VALIDATION)
3. Maker-Checker: validation demande par superviseur
4. Si validé → PHASE 2; Si rejeté → notification motif au client

PHASE 2: ANALYSE
1. Agent crédit analyse dossier (credit-analysis.php)
2. Scoring automatique (credit-scoring.php): score sur 1000
3. Si score ≥ 700: recommandé
4. Si score 500-700: comité requis
5. Si score < 500: refus recommandé

PHASE 3: COMITÉ (si requis)
1. Convocations membres comité (credit-committee.php)
2. Saisie avis individuels
3. Décision collégiale: ACCORDÉ / REFUSÉ / CONDITIONNEL
4. Si conditionnel: conditions spécifiques ajoutées au dossier

PHASE 4: DÉCISION
1. Saisie décision finale (credit-decision.php)
2. Si accordé: montant, taux, durée, conditions particulières
3. Si refusé: motif envoyé au client
4. Validation finale (Maker-Checker)

PHASE 5: CONTRAT
1. Génération contrat (credit-contract.php): PDF avec toutes conditions
2. Signature client (signature-pad ou papier scanné)
3. Archivage contrat signé

PHASE 6: DÉCAISSEMENT
1. Vérification conditions suspensives levées
2. Décaissement (credit-disbursement.php): montant vers compte client
3. Frais de dossier prélevés
4. Écriture comptable: débit compte prêt, crédit compte client
5. Échéancier généré
6. Notification client (SMS + email)

PHASE 7: REMBOURSEMENT
1. Échéance auto: prélèvement compte épargne/courant (si autorisé)
2. Remboursement manuel: guichetier ou agent terrain
3. Calcul intérêts: méthode actuarielle
4. Si impayé > 5 jours: relance automatique (SMS)
5. Si impayé > 30 jours: statut IMPAYÉ, provisions
6. Si impayé > 90 jours: contentieux, poursuites

5.4 WORKFLOW CAISSE (OUVERTURE → OPÉRATIONS → FERMETURE)
----------------------------------------------------------------
1. Guichetier ouvre session (cash-open.php):
   - Identifie caisse
   - Saisit solde initial (XOF)
   - Billetage initial
   - Signature
2. Session active: opérations (dépôts, retraits, virements)
3. Chaque opération enregistrée dans journal caisse
4. En fin de journée: fermeture caisse (cash-close.php):
   - Solde théorique = ouverture + dépôts - retraits
   - Billetage physique
   - Solde réel
   - Calcul écart
   - Si écart ≠ 0: motif + validation superviseur si > seuil
   - Signature fermeture
5. Session fermée: CLOS ou PENDING_VALIDATION

5.5 WORKFLOW MAKER-CHECKER GÉNÉRIQUE
----------------------------------------------------------------
1. MAKER: Crée/modifie entité (client, compte, crédit, etc.)
2. MAKER: Soumet pour validation (statut → PENDING_APPROVAL)
3. Notifications envoyées aux validateurs (superviseurs, managers)
4. CHECKER: Consulte file d'attente (approval-queue.php)
5. CHECKER: Examine détails, changements, pièces
6. CHECKER: Actions possibles:
   a. APPROUVER: entité validée, actions exécutées, notification maker
   b. REJETER: entité refusée, motif loggué, notification maker
   c. RETOUR POUR CORRECTION: entité retournée au maker, commentaire
7. Maker reçoit notification et peut corriger/resoumettre

RÈGLES:
- Maker et Checker ne peuvent pas être la même personne
- Checker doit avoir le rôle SUPERVISEUR ou supérieur
- Double validation pour montants > 10M FCFA
- SLA: P0 4h, P1 8h, P2 24h, P3 48h
- Escalade automatique si SLA dépassé

5.6 WORKFLOW KYC
----------------------------------------------------------------
1. Client créé (statut: KYC_PENDING)
2. Checklist KYC générée automatiquement selon type client:
   - Physique: CNI/passeport, selfie, justificatif domicile, déclaration BÉFI
   - Moral: statuts, RCCM, CNPS, représentants légaux
3. Guichetier/Agent upload documents
4. Vérification documents par conformité:
   - Valide: documents conformes, statut → KYC_VALIDATED
   - Invalide: documents rejetés, motif, statut → KYC_REJECTED
   - Complément: documents supplémentaires requis
5. Client notifié du statut KYC
6. Screening PEP et sanctions (automatique)
7. Profil risque client défini (faible/moyen/élevé)
8. KYC à renouveler: 1 an (risque élevé), 3 ans (moyen), 5 ans (faible)

5.7 WORKFLOW EOD (END OF DAY)
----------------------------------------------------------------
PRÉ-REQUIS: Toutes caisses fermées, toutes opérations validées
1. Déclenchement: automatique (horaire configurable) ou manuel
2. Vérification pré-requis (auto-check)
3. Étape 1: Arrêt des opérations (plus aucune transaction acceptée)
4. Étape 2: Calcul intérêts épargne (si dernier jour période)
5. Étape 3: Calcul provisions crédit
6. Étape 4: Rapprochement caisse/vault
7. Étape 5: Écritures comptables EOD
8. Étape 6: Génération bordereaux et rapports
9. Étape 7: Archivage des journaux
10. Étape 8: Ouverture nouveau jour comptable
MONITORING: Barre progression, statut chaque étape, logs temps réel
SI ÉCHEC: Pause, afficher cause, permettre reprise ou forcer contournement

5.8 WORKFLOW COMPTABLE (CLÔTURE DE PÉRIODE)
----------------------------------------------------------------
1. Vérification: toutes écritures validées, aucun brouillard en cours
2. Calcul amortissements et provisions
3. Inventaire stocks (si actif)
4. Lettrage comptes (automatique + manuel si anomalies)
5. Balance de vérification (doit être équilibrée)
6. États financiers: Bilan, CGR, Tableau flux
7. Clôture exercice: extinction comptes 6/7, affectation résultat
8. Ouverture nouvel exercice: reports à nouveau
9. Archivage exercice clôturé (lecture seule)

5.9 WORKFLOW MULTI-AGENCE (OPÉRATION DÉPLACÉE)
----------------------------------------------------------------
1. Client se présente dans agence B (domicilié à agence A)
2. Guichetier agence B initie opération déplacée
3. Vérification: client existe, compte actif
4. Opération marquée "déplacée" avec agence origine
5. Frais inter-agence appliqués (si applicable)
6. Exécution locale (débit/crédit immédiat)
7. Rapprochement inter-agence nocturne
8. Écritures comptables: compte transit agence B → compte transit agence A
9. Compensation réalisée

7. EXIGENCES UX SPÉCIFIQUES
================================================================================

7.1 RESPONSIVE
- Desktop (>1280px): Sidebar expanded 256px, content 12-col grid
- Tablet (768-1279px): Sidebar collapsed 72px (icônes), content full width
- Mobile (<768px): Sidebar off-canvas (hamburger menu), stacked layouts, simplified tables

7.2 COMPORTEMENTS GÉNÉRAUX
- Keyboard shortcuts: Ctrl+S (save), Ctrl+Enter (submit), Ctrl+K (global search), Escape (close modal)
- Autosave brouillon: tous les 60s pour les formulaires longs
- Debounce recherche: 300ms
- Infinite scroll: pas de pagination, remplacer par "Charger plus" bouton
- Tooltips sur icônes (title attribute minimum)
- Confirm dialogs pour toutes actions destructrices

7.3 ACCESSIBILITÉ
- Contrast ratio minimum 4.5:1 (WCAG AA)
- Focus visible sur tous les éléments interactifs
- Labels ARIA pour icônes seules
- Keyboard navigation (Tab, Enter, Escape)
- Screen reader: aria-live regions pour mises à jour dynamiques
- headings hiérarchiques (h1 → h2 → h3)

7.4 GESTION DES ERREURS
- Validation inline: erreur sous chaque champ immédiatement après blur
- Error summary: en haut du formulaire pour les erreurs de soumission
- Toast notifications pour succès/erreur (auto-dismiss 5s)
- HTTP errors: 401→login, 403→page interdite, 404→page non trouvée, 500→page d'erreur
- Network error: bannière "Connexion perdue" + retry automatique
- Empty states: illustration + message + action (ex: "Aucun client. Créer le premier client")

7.5 CHARGEMENT
- Skeleton screens pour tableaux (shimmer rows)
- Spinner pour actions AJAX
- Progress bar pour uploads
- "Page en cours de chargement..." pour navigation complète (Turbo)

7.6 ÉTATS PAR DÉFAUT
- Loading: skeleton UI, shimmer animation, pas de spinner centré
- Empty: illustration personnalisée + message + CTA
- Error: message clair + action possible (retry, go back)
- Success: toast notification + redirect or confirmation screen
