Voici la **définition complète de tous les écrans du projet MicroFin Core Banking**, en couvrant :

1. **application web opérateur** ;
2. **portail client web** ;
3. **application mobile client** ;
4. **application mobile agent terrain** ;
5. **USSD** ;
6. **dashboard agent banking** ;
7. **portail développeur Open Banking** ;
8. **console communication, SMS, WhatsApp, chatbot** ;
9. **interfaces techniques et administratives**.

Le dossier technique existant prévoit déjà un inventaire initial des écrans pour le dashboard opérateur, le portail e-banking et l’application mobile . Les documents complémentaires indiquent aussi que l’application mobile actuelle doit être étendue vers une vraie application native, une application agent terrain, un serveur USSD, un dashboard agent banking, un portail API et une console de communication . L’inventaire ci-dessous reprend donc l’existant, puis l’étend pour couvrir tous les modules du PRD.

# 1. Principes généraux d’interface à respecter

## 1.1 Architecture des interfaces

Toutes les interfaces doivent être construites autour d’une logique commune :

| Élément           | Règle                                                                                                                |
| ----------------- | -------------------------------------------------------------------------------------------------------------------- |
| Identité visuelle | Couleurs MicroFin, interface sobre, bancaire, claire, professionnelle.                                               |
| Navigation        | Menus par rôle, aucune fonction visible si l’utilisateur n’a pas les droits.                                         |
| Sécurité          | Authentification forte sur les actions sensibles.                                                                    |
| Traçabilité       | Toute action importante doit afficher l’utilisateur, l’agence, la date métier et le statut.                          |
| Maker-Checker     | Toute action sensible passe par une file de validation.                                                              |
| Recherche         | Chaque grand module doit avoir recherche, filtres, tri, pagination et export.                                        |
| Consultation      | Chaque entité importante doit avoir une fiche détaillée.                                                             |
| Historique        | Chaque fiche doit afficher les événements, modifications et validations.                                             |
| Documents         | Les documents générés doivent être consultables, téléchargeables et archivés.                                        |
| Responsive        | Les écrans web doivent être utilisables sur ordinateur et tablette.                                                  |
| Mobile            | Les applications mobiles doivent privilégier les actions rapides, les cartes synthétiques et les formulaires courts. |

## 1.2 Structure commune des écrans web internes

Chaque écran web opérateur doit avoir :

| Zone             | Contenu                                                               |
| ---------------- | --------------------------------------------------------------------- |
| Sidebar gauche   | Modules accessibles selon le rôle.                                    |
| Topbar           | Recherche globale, notifications, profil, agence active, date métier. |
| Fil d’Ariane     | Position de l’utilisateur dans le module.                             |
| Titre de page    | Nom de l’écran et description courte.                                 |
| Zone actions     | Boutons principaux : créer, exporter, imprimer, valider, rejeter.     |
| Zone filtres     | Filtres par agence, statut, date, produit, utilisateur, montant.      |
| Zone données     | Tableau, cartes, graphique ou formulaire.                             |
| Panneau latéral  | Détail rapide, historique, validation ou commentaires.                |
| Pied fonctionnel | Pagination, total, date de dernière mise à jour.                      |

## 1.3 Structure commune des écrans mobiles

L’application mobile client doit respecter la navigation déjà prévue par le dossier technique : **Accueil, Comptes, Opérations, Notifications, Plus** .

| Zone              | Contenu                                                    |
| ----------------- | ---------------------------------------------------------- |
| Header            | Nom du client, avatar, icône notification.                 |
| Corps             | Carte principale, actions rapides, listes courtes.         |
| Bottom navigation | Accueil, Comptes, Opérations, Notifications, Plus.         |
| Action principale | Bouton flottant pour virement, dépôt, paiement ou demande. |
| Sécurité          | Code PIN, biométrie, OTP selon le niveau de risque.        |
