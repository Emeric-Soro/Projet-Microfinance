1. SYSTÈMES D'INFORMATION BANCAIRE
================================================================================

  1.1 Architecture générale
  ──────────────────────────
  - Architecture 5 couches : Controller → DTO → Service → Repository → Entity
  - Base de données Oracle XE 21c
  - Migrations Flyway (28 fichiers : B0 + V1 à V27)
  - API REST documentée via SpringDoc OpenAPI (Swagger UI)
  - Sécurité JWT stateless (HS256)
  - Cache Caffeine pour les paramètres de tarification
  - Spring Batch pour les traitements EOD (End of Day)
  - AOP pour l'audit (@AuditLog annotation)
  - Injection de dépendances par constructeur (Lombok @RequiredArgsConstructor)
  - MapStruct pour le mapping DTO <-> Entité
  - Gestion des exceptions centralisée (GlobalExceptionHandler)

  1.2 Modules fonctionnels couverts
  ─────────────────────────────────
  - Core bancaire : Clients, Comptes, Transactions
  - Crédit : Demandes, Produits, Garanties, Échéances, Remboursements
  - Épargne : Produits, Opérations, Dépôts à terme
  - Caisse / Coffre : Sessions, Approvisionnement, Délestage
  - Comptabilité : Partie double, Schémas, Journaux, Écritures, Clôture
  - Conformité : Sanctions, PEP, AML, BIC, Déclarations fiscales
  - Paiements : Mobile Money, Cartes Visa, Chéquiers, Virements externes
  - Scoring : Grilles, Critères, Résultats
  - Risques : Stress tests, Incidents
  - Trésorerie : Budget, Commandes, Fournisseurs, Immobilisations, Paie
  - Organisation : Régions, Agences, Guichets, Employés, Mutations
  - Sécurité : JWT, RBAC, Permissions, Audit, Rate Limiting
  - Digital : Mobile Money, Wallets, Open Banking
  - Tontines : Cotisations, Tours
  - Commissions : Agent, Inter-agence
  - Batch : EOD (stubbé — à implémenter)
  - RH : Employés, Mutations, Paie
  - Achats : Commandes, Fournisseurs
  - Immobilisations : Amortissements, Inventaire
  - Agents Banking : Portefeuille, Commissions
  - Cartes : Plafonds, Statuts
  - Chéquiers : Demandes, Validations, Livraisons
  - Fiscalité : Déclarations, Rapports prudentiels
  - Audit : Traçabilité des actions, SystemAuditLog

  1.3 Infrastructure technique
  ────────────────────────────
  - Docker Compose : Oracle XE + Backend Spring Boot + CloudBeaver
  - Maven Wrapper (mvnw) — build sans Maven installé
  - Variables d'environnement externalisées (.env)
  - Healthchecks Docker pour l'orchestration
  - CloudBeaver pour l'administration DB (UI web)
  - 41 fichiers de test (JUnit 5 + Testcontainers + Mockito)

  4.3 Fonctionnalités
  ───────────────────
  - Dépôt sur compte (POST /api/transactions/depot)
  - Retrait depuis compte (POST /api/transactions/retrait)
  - Virement interne (POST /api/transactions/virement)
  - Validation 4-yeux (POST /api/transactions/{id}/validation)
  - Rejet de transaction (POST /api/transactions/{id}/rejet)

  - Idempotency key pour les opérations externes

  4.4 Workflow Transaction Complet
  ────────────────────────────────
  [Workflow Dépôt]
  1. Agent saisit le montant et le compte destination
  2. Validation : montant positif, session caisse ouverte
  3. Vérification supervision : si montant >= 500 000 FCFA → 4-yeux
  4. Création transaction (statut EN_ATTENTE)
  5. Si supervision requise :
      a. Soumission au superviseur
      b. Superviseur valide ou rejette
      c. Anti-auto-validation (initiateur != superviseur)
  6. Vérification comptable (verifierTransactionComptable)
  7. Exécution :
      a. Crédit du compte destination
      b. Débit des frais (compte de charges)
      c. Mise à jour solde caisse
  8. Comptabilisation (comptabiliserTransaction)
  9. Analyse conformité
  10. Notification si virement

  [Workflow Retrait]
  1. Agent saisit le montant et le compte source
  2. Vérification fonds disponibles (solde >= montant + frais)
  3. Idem workflow validation 4-yeux si >= 500 000 FCFA
  4. Exécution : débit du compte, crédit des frais, mise à jour caisse

  [Workflow Virement]
  1. Agent saisit montant, compte source, compte destination
  2. Vérification fonds disponibles
  3. Vérification BIC si externe
  4. Idem workflow validation 4-yeux si >= 500 000 FCFA
  5. Exécution : débit source, crédit destination, frais
  6. Publication de l'événement VirementEffectueEvent

  [Workflow Extourne]
  1. Transaction originale identifiée
  2. Création d'une contre-opération (opération inverse)
  3. Types supportés : DEPOT_CASH, RETRAIT_CASH, VIREMENT_INTERNE,
     CREDIT_DEBLOCAGE, MOBILEMONEY_CASHIN, MOBILEMONEY_CASHOUT
  4. Comptabilisation de l'extourne

  - SensEcriture.java (enum) — DEBIT, CREDIT

  5.2 Services
  ────────────
  - ComptabiliteExtensionService.java (867+ lignes)
      - Bootstrap automatique (classes, comptes, journaux, schémas)
      - Génération d'écritures par schéma
      - Clôture comptable
      - Balance générale
      - Grand Livre
  - DoubleEntryServiceImpl.java
      - Validation équilibre Débit = Crédit
      - Contrôle d'intégrité
  - JournalEntryServiceImpl.java
      - Gestion des journaux

  5.3 Fonctionnalités comptables
  ──────────────────────────────
  - Comptabilité en partie double complète
  - Bootstrap automatique au démarrage :
      - 5 classes comptables
      - ~20 comptes comptables (Caisse, Banque, Clients, etc.)
      - 7 journaux (ACH, BAN, CAI, OPE, ECH, ANL, DIV)
      - ~25 schémas comptables
  - Génération d'écritures basée sur les schémas pour :
      - Dépôt cash, Retrait cash, Virement interne
      - Déblocage crédit, Remboursement crédit
      - Mobile Money CashIn/Out
      - Agios, Frais, Intérêts, Provisions
  - Validation équilibre (total Débit = total Crédit)
  - Clôture comptable (mensuelle, annuelle)
  - Balance générale
  - Grand Livre
  - Rapport de balance par compte
  - Calcul des soldes comptables

  5.4 Workflow Comptable
  ──────────────────────
  [Bootstrap Comptable (au démarrage)]
  1. Création des 5 classes comptables
  2. Création des comptes comptables
  3. Création des journaux
  4. Création des schémas comptables

  [Écriture Comptable]
  1. Déclenchement par une opération bancaire
  2. Sélection du schéma comptable adapté
  3. Génération des lignes Débit/Crédit
  4. Validation de l'équilibre (total Débit >= total Crédit)
  5. Enregistrement dans le journal approprié
  6. Mise à jour des soldes comptables

  [Clôture Comptable]
  1. Vérification : aucune transaction en attente
  2. Vérification : équilibre comptable
  3. Calcul et comptabilisation des agios
  4. Calcul et comptabilisation des provisions
  5. Détection et comptabilisation des impayés
  6. Traitement EOD
  7. Clôture de la période
  8. Ouverture de la nouvelle période

  3. Décision (Maker-Checker) :
      a. Création de l'action "DecisionDemandeCredit"
      b. Validation par superviseur

      b. Validation par superviseur
      c. Génération de l'échéancier

      a. Paiement des échéances
      b. Mise à jour du solde du crédit
  6. Détection impayés (automatique, EOD)
  7. Provisions périodiques

  8.3 Workflow Caisse
  ───────────────────
  [Session de Caisse]
  1. Ouverture :
      a. Agent soumet demande d'ouverture (Maker)
      b. Superviseur valide (Checker)
      c. Solde d'ouverture enregistré
  2. Opérations (dépôts, retraits) :
      a. Chaque opération met à jour le solde théorique
      b. Suivi en temps réel
  3. Fermeture :
      a. Agent soumet demande de fermeture (Maker)
      b. Comptage physique vs solde théorique
      c. Superviseur valide (Checker)

  30.4 Workflow Maker-Checker
  ────────────────────────────
  1. Utilisateur A (Maker) soumet une action
      → ActionEnAttente créée (statut: EN_ATTENTE)
  2. Utilisateur B (Checker) consulte les actions en attente
  3. Utilisateur B valide ou rejette
      → Anti-auto-validation : A != B
      → ValidationExtensionService vérifie les statuts
  4. Si validée :
      a. PendingActionExecutionService exécute le handler
      b. L'action est marquée comme exécutée
  5. Si rejetée :
      a. L'action est marquée comme rejetée
      b. Motif du rejet enregistré

   - Caissier → SessionCaisseController → PendingActionSubmissionService

   - Fermeture → Vérification écart → Validation

7. Séquence — Audit (Tracabilité)
   - Méthode annotée → AuditLogAspect (AOP)
   - Capture arguments → Création SystemAuditLog → Persistance

   - Spring Boot Scheduler → EodBatchConfig → JobLauncher
   - Step 1→2→3→4→5→6→7 séquentiellement
   - Chaque étape → Tasklet correspondant → Log + contexte
